## 📑 프로젝트 규칙

### Branch Strategy
> - main 브랜치 기본 생성 (default)
> - main 브랜치로 push 제한
> - PR의 merge가 되기 전 최소 1인 이상 approve 필수

### Git Convention
> 1. 적절한 커밋 접두사 작성
> 2. 커밋 메시지 내용 작성
> 3. 내용 뒤에 이슈 (#이슈 번호)와 같이 작성하여 이슈 연결

> | 접두사        | 설명                           |
> | ------------- | ------------------------------ |
> | ✨ feat :     | 새로운 기능 구현               |
> | 🔧 fix :      | 기능 수정, 버그 수정            
> | 💡 chore :      | 오타 수정 및 새로운 기능이 추가되지 않고, 코드가 변경 된 경우 (주석 추가 및 수정 포함)|
> | 📝 docs     | 문서 추가 및 수정              |
> | 🚚 build :   | 빌드 관련 파일 수정 및 삭제한 경우|
> | ✅ test :  | 테스트 코드, 리팩터링 테스트 코드 추가(프로덕션 코드 변경 X) |
> | ♻️ refactor :| 코드 리팩터링,     |
> | 🚨 hotfix :    | 긴급 수정, 긴급 오류 점검 등 |
> | 🏷️ rename : | 파일 이름을 수정한 경우               |


### Pull Request
> ### Title
> * [working branch name] 작업내용 간략한 설명'과 같이 작성합니다.
> * **example** : [Feat/user-model] 유저 모델 구성, 마이그레이션 생성

> ### PR Type
  > - [ ] FEAT: 새로운 기능 구현
  > - [ ] FIX: 버그 수정
  > - [ ] DOCS: 문서 추가 및 수정
  > - [ ] REFACTOR: 코드 리팩토링
  > - [ ] TEST: 테스트 관련
  > - [ ] DEPLOY: 배포 관련
  > - [ ] CONF: 빌드, 환경 설정
  > - [ ] CHORE: 기타 작업

> ### Description
> * 구체적인 작업 내용을 작성해주세요.
> * 이미지를 별도로 첨부하면 더 좋습니다 👍

> ### Discussion
> * 추후 논의할 점에 대해 작성해주세요.

### Code Convention
> ## Backend
> - 패키지명 전체 소문자
> - 클래스명, 인터페이스명 CamelCase
> - 상수명 SNAKE_CASE
> - test 케이스 메서드는 접두사로 test_ 사용(ex. test_user_create)

> ## Frontend
> - styled-Component 변수명 S + 변수명 (ex. Swrap)
> - styled-Component는 return문 위에 작성
> - 크게는 styled-Component, 그 안에서 className 사용 
> - Event handler 사용 (ex. handle ~)
> - export방식 (ex. export default ~)
> - 화살표 함수 사용

### Communication Rules
> - 정규시간은 zep 활용
> - Discord 활용 (공지사항, git alarm 확인)
> - 정기 회의(매주 월요일)
> - 정기 브리핑(매일 오전 10시 10분)
